# Gamificación y Experiencia del Usuario

Esta sección define la filosofía de gamificación de CodeQuest.

La prioridad del proyecto NO es enseñar SQL.

La prioridad es conseguir que el usuario quiera regresar todos los días.

Toda nueva funcionalidad debe evaluarse preguntando:

"¿Esto aumenta la motivación del usuario para volver mañana?"

Si la respuesta es no, debe replantearse.

---

# Principios de Diseño

El aprendizaje debe sentirse como una consecuencia natural de jugar.

Nunca como una obligación.

Cada interacción debe generar una sensación positiva.

El usuario siempre debe percibir progreso.

Nunca debe sentirse perdido.

Nunca debe sentirse castigado.

---

# Octalysis Framework

Toda nueva funcionalidad debe apoyar uno o varios de estos impulsores psicológicos.

## 1. Epic Meaning

El usuario no está haciendo ejercicios.

Está ayudando a Byte a restaurar una red digital de conocimiento.

Cada curso representa una región del mundo.

Cada unidad recupera parte de ese mundo.

Cada avance tiene significado.

---

## 2. Development & Accomplishment

El usuario debe percibir progreso constantemente.

Ejemplos:

* XP
* niveles
* insignias
* porcentaje
* barras
* regiones desbloqueadas
* logros

Nunca ocultar el progreso.

Siempre mostrarlo.

---

## 3. Empowerment of Creativity

Permitir experimentar.

Especialmente durante las prácticas.

El usuario debe poder probar consultas SQL sin miedo.

Las respuestas incorrectas deben enseñar.

Nunca castigar.

---

## 4. Ownership

El usuario debe sentir que su progreso le pertenece.

Mostrar:

* perfil
* estadísticas
* historial
* colecciones
* logros
* progreso global

Todo debe reforzar esa sensación.

---

## 5. Social Influence

Diseñar la arquitectura para permitir futuras funciones sociales.

Ejemplos:

* compartir logros
* comparar progreso
* rankings opcionales
* desafíos entre amigos

No es necesario implementarlo ahora.

Pero la arquitectura debe permitirlo.

---

## 6. Scarcity

No mostrar todo el contenido desde el inicio.

Desbloquear regiones gradualmente.

Los nuevos mundos deben sentirse especiales.

---

## 7. Curiosity

Siempre debe existir algo interesante por descubrir.

Ejemplos:

* nuevo mundo
* nueva habilidad
* Byte cambia de expresión
* cofres
* recompensas
* eventos

El usuario nunca debe sentir que ya vio todo.

---

## 8. Loss & Avoidance

Utilizar este principio con moderación.

Nunca manipular.

Nunca castigar excesivamente.

Puede utilizarse mediante:

* mantener la racha
* recordar objetivos diarios
* eventos limitados

El objetivo es motivar.

No generar ansiedad.

---

# Feedback inmediato

Toda acción importante debe producir una respuesta inmediata.

Ejemplos:

Responder correctamente:

* sonido
* animación
* Byte celebra
* XP aparece
* barra avanza

Responder incorrectamente:

* explicación inmediata
* Byte anima a continuar
* posibilidad de volver a intentar

Nunca dejar una acción sin respuesta.

---

# Sistema de recompensas

Toda recompensa debe tener significado.

Utilizar:

XP

Niveles

Insignias

Cristales

Artefactos

Nuevas regiones

Elementos cosméticos

No utilizar recompensas únicamente decorativas.

Cada una debe reforzar la motivación.

---

# XP

El XP representa experiencia adquirida.

Nunca debe regalarse sin esfuerzo.

Debe otorgarse por:

* completar bloques
* prácticas
* desafíos
* rachas
* exploración

Mostrar siempre cuánto XP se obtuvo.

---

# Niveles

Cada nivel representa el crecimiento del explorador.

El nivel nunca debe subir sin una pequeña celebración.

---

# Rachas

Las rachas deben motivar.

Nunca generar culpa.

Si el usuario pierde una racha:

Mostrar un mensaje positivo.

Invitar a comenzar una nueva.

Nunca hacer sentir fracaso.

---

# Byte

Byte es el compañero del usuario.

Nunca un profesor.

Nunca un jefe.

Nunca un asistente.

Su papel es explorar junto al jugador.

Siempre celebra.

Siempre anima.

Nunca critica.

Nunca regaña.

---

# Narrativa

Todo CodeQuest ocurre dentro de un universo digital.

El usuario ayuda a reconstruir una red de conocimiento.

Cada región representa un área distinta de la programación.

Cada nuevo curso es un nuevo territorio.

Cada práctica importante representa un desafío.

Cada examen representa la recuperación completa de una región.

La narrativa debe reforzar constantemente la sensación de aventura.

---

# Mundo Vivo

El mundo debe reaccionar al progreso.

Ejemplos:

El camino se ilumina.

Byte avanza.

Se desbloquean nuevas zonas.

Aparecen recompensas.

Los circuitos cobran vida.

El usuario debe sentir que realmente está cambiando el mundo.

---

# Regla de Diseño

Nunca agregar una funcionalidad únicamente porque sea común en otras plataformas.

Cada nueva mecánica debe responder claramente:

¿Qué emoción busca provocar?

¿Qué impulsor psicológico del Octalysis fortalece?

¿Cómo contribuye a que el usuario quiera regresar mañana?

Si no existe una respuesta clara, la funcionalidad debe reconsiderarse.
