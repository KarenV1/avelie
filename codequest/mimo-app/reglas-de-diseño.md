# CodeQuest - Permanent Project Instructions

Este documento contiene las reglas permanentes del proyecto.

Todas las decisiones aquí descritas son parte de la identidad de CodeQuest.

No deben modificarse a menos que el usuario lo solicite explícitamente.

Nunca improvisar una alternativa si una regla ya existe en este documento.

---

# Filosofía del proyecto

CodeQuest NO es una plataforma tradicional de cursos.

CodeQuest es un videojuego de aprendizaje.

El usuario no siente que está estudiando.

El usuario siente que explora un mundo digital mientras aprende programación.

Todas las decisiones de diseño deben reforzar esa sensación.

---

# Identidad visual

La aplicación utiliza una identidad minimalista.

No utilizar colores llamativos.

El color principal es el amarillo CodeQuest.

El negro domina completamente la interfaz.

El amarillo únicamente sirve para comunicar:

* progreso
* interacción
* logros
* recompensas
* elementos activos
* Byte

Mientras menos amarillo exista, mayor impacto tendrá cuando aparezca.

---

# Estilo general

La interfaz debe sentirse:

* moderna
* premium
* limpia
* tecnológica
* minimalista
* elegante

Nunca debe parecer infantil.

Nunca debe parecer una copia de Duolingo.

Nunca debe parecer una aplicación educativa tradicional.

La referencia es una interfaz de videojuego.

---

# Inspiración

Inspirarse únicamente en la experiencia de exploración de videojuegos como:

* Hollow Knight
* Ori
* Zelda
* Pokémon
* Animal Crossing
* Mario Wonder

NO copiar personajes.

NO copiar escenarios.

NO copiar estilos gráficos.

Únicamente inspirarse en:

* sensación de aventura
* progresión
* curiosidad
* exploración

---

# Filosofía del mapa

El mapa NO es una lista.

El mapa representa un mundo digital.

Cada unidad representa una región.

Cada bloque representa un descubrimiento.

Cada práctica representa un desafío.

Cada examen representa un jefe.

El usuario debe sentir curiosidad por descubrir el siguiente nodo.

---

# Camino

El camino es uno de los elementos más importantes de la interfaz.

No utilizar listas verticales.

No utilizar tarjetas apiladas.

El recorrido debe serpentear ligeramente.

Debe sentirse natural.

El camino representa una red digital.

No una carretera.

No un sendero físico.

Debe parecer una conexión de datos.

---

# Progreso

El progreso debe entenderse incluso sin leer números.

El recorrido completado debe iluminarse.

Los nodos completados cambian visualmente.

Byte siempre indica la posición actual del usuario.

---

# Byte

Byte es la mascota oficial de CodeQuest.

Nunca debe rediseñarse.

Nunca deben cambiarse sus proporciones.

Nunca deben modificarse:

* ojos
* cuerpo
* patas
* antenas
* colores
* personalidad

Si se requiere una nueva ilustración, SIEMPRE debe respetar el diseño oficial existente.

No improvisar nuevas versiones.

No reinterpretar el personaje.

No simplificarlo.

No estilizarlo.

No convertirlo en robot.

No convertirlo en insecto.

No cambiar su silueta.

Byte es una criatura digital.

No una inteligencia artificial.

No un robot.

No un insecto.

---

# Personalidad de Byte

Byte es:

* curioso
* amigable
* silencioso
* optimista
* juguetón
* compañero

Nunca habla demasiado.

Nunca roba protagonismo a la interfaz.

Su función es acompañar al usuario.

No ser un asistente virtual.

---

# Animaciones de Byte

Las animaciones siempre deben ser discretas.

Permitidas:

* respiración
* flotación
* parpadeo
* movimiento de antenas
* pequeñas miradas
* estela digital
* partículas muy suaves

Nunca hacer animaciones exageradas.

Nunca distraer al usuario.

---

# Fondo

El fondo debe permanecer oscuro.

La cuadrícula forma parte de la identidad.

Debe ser extremadamente sutil.

Nunca competir con el contenido.

---

# Microanimaciones

Toda la interfaz debe sentirse viva.

Agregar movimiento únicamente donde aporte valor.

Ejemplos:

* barras de progreso
* nodos activos
* Byte
* recompensas
* circuitos
* desbloqueo de unidades

Nunca abusar de las animaciones.

---

# Arquitectura

Siempre reutilizar componentes existentes antes de crear nuevos.

Mantener la arquitectura React actual.

Evitar duplicación de componentes.

Crear componentes reutilizables.

No romper la estructura del proyecto.

---

# Diseño responsive

Toda pantalla debe funcionar correctamente en:

* móvil
* tablet
* escritorio

El diseño mobile-first tiene prioridad.

---

# Código

Priorizar:

* legibilidad
* escalabilidad
* mantenibilidad
* componentes reutilizables
* nombres descriptivos

Evitar código duplicado.

Evitar soluciones rápidas que dificulten futuras mejoras.

---

# Regla principal

Cuando exista una duda de diseño, no improvisar.

Consultar primero las reglas descritas en este documento.

Si una decisión no está documentada, mantener coherencia con la identidad existente del proyecto.

La consistencia del producto es más importante que introducir nuevas ideas.
